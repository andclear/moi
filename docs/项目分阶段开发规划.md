# 《回音 / Echo》项目分阶段开发规划

## 一、规划目标

本文档用于指导《回音 / Echo》的实际开发过程。开发时以 `docs/项目开发文档.md` 为产品与工程总规范，以本文档为阶段执行顺序。

开发过程必须遵守以下原则：
- 先完成工程底座，再开发业务页面。
- 先完成本地数据闭环，再接入完整 AI 生成链路。
- 先完成主角角色卡生成闭环，再开发第二阶段增强功能。
- 所有 AI 调用必须经过统一 LLM 适配层。
- 所有本地持久化必须经过 Dexie repository 或领域 service。
- 页面组件只做组合，业务逻辑放入 `features`。
- 第一阶段不开发美化代码生成、侦探助手笔记、关系网与配角、侧写报告。
- 项目包管理器固定使用 pnpm。
- 产品文案、项目文档、代码注释使用简体中文。

### 阶段 3：角色档案系统

**目标**：实现全局角色档案这个产品事实中心。

**实现功能**：
- 实现 `dossierStore`：
  - 当前 Markdown 内容。
  - 当前块级元数据。
  - 锁定状态。
  - 用户确认事实状态。
  - 自动保存状态。
- 实现 Markdown 档案结构：
  - `## 核心人格`
  - `## 外貌特征`
  - `## 背景故事`
  - `## 核心矛盾`
  - `## 说话风格`
  - `## 世界观`
  - `## 开场白`
- 实现块级元数据映射：
  - `blockId`
  - `section`
  - `contentHash`
  - `source`
  - `locked`
  - `updatedByGenerationId`
  - `updatedAt`
- 实现 `DossierPanel`。
- 实现 `DossierEditor`。
- 实现字段或段落锁定。
- 实现用户编辑后自动标记为 `user_confirmed`。
- 实现 AI 推测内容标记为 `ai_inferred`。
- 实现 Markdown 渲染预览。

**交付成果**：
- 用户可以查看、编辑、锁定角色档案。
- 用户编辑内容会自动保存。
- 用户编辑块会标记为用户确认事实。
- 锁定块不会被后续 AI 更新覆盖。
- 角色档案侧栏可被所有主流程页面复用。

**验收标准**：
- 编辑 Markdown 后刷新页面，内容保持一致。
- 锁定某个块后，模拟 AI 更新不会覆盖该块。
- 用户手动修改某个块后，该块元数据变为 `user_confirmed`。
- `DossierPanel` 在所有主流程步骤可见。

## 五、第一阶段不可偏离事项

第一阶段开发过程中不得加入以下功能：
- 美化代码生成。
- 正则绑定器。
- 侦探助手笔记完整面板。
- 关系网。
- 配角微型寻人。
- 侧写报告导出。
- 多项目并行编辑。
- 搜索、标签、排序。
- 复杂实时聊天终审。

第一阶段开发过程中必须保持以下约束：
- 角色档案是 Markdown 编辑主体。
- 用户确认事实必须进入后续提示词。
- 锁定内容不能被 AI 覆盖。
- 所有 AI 调用必须记录 `GenerationTask`。
- 所有项目数据必须进入 IndexedDB。
- 所有导出内容最终进入单个 JSON。
- PNG 写入内容必须是格式化 JSON。
- PNG 角色卡导出必须使用用户上传图片作为载体。
- 上传图片先转换为 PNG，再写入 `tEXt` 数据块。
- 预置密钥不能进入前端。
- 页面不得直接调用 OpenAI SDK。
- 页面不得直接操作 Dexie 表。
- 提示词不得散落在页面组件中。
