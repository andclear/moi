# 《来岛上 / MOI》项目开发文档

## 一、项目概述

**项目名称**：来岛上 / MOI  
**项目类型**：前端优先的网页工具  
**部署目标**：Vercel  
**后端数据库**：Neon PostgreSQL  
**包管理器**：pnpm  
**目标用户**：SillyTavern 等 AI 角色扮演社区的创作者，尤其是缺乏角色卡编写经验的新手  
**核心价值**：将“创建角色卡”的过程重新设计为一场“侦探寻人”的沉浸式互动。用户通过自然语言描述理想角色，系统以侦探搭档的身份，通过多轮提问、反例排除、叙事碎片选择、日记破译、世界书生成、开场白生成和终审对话测试，帮助用户逐步锁定角色人格、世界观与开场语境，并最终导出严格对齐 Character Card V3 的角色卡 JSON，以及写入 PNG `tEXt` 数据块的角色卡图片。

**设计哲学**：
- **寻找，而非创建**：保留社区“角色是本就存在，等待被发现”的浪漫想象。
- **零门槛**：新手用户仅需用自然语言说几句话，即可完成专业级角色卡草案。
- **强掌控**：所有 AI 生成内容都沉淀到全局角色档案中，用户可随时修改、锁定，并直接影响后续生成。
- **连续性**：主流程在一个侦探档案室式工作台内连续展开，通过无感路由切换和固定布局减少打断。
- **本地优先与隐私安全**：用户自配 API Key 的模式下，所有配置与创作数据均保存在浏览器本地。预置服务模式需通过激活码控制使用权，并避免在浏览器中暴露真实服务密钥。

**语言约束**：
- 产品界面文案使用简体中文。
- 项目文档使用简体中文。
- 代码注释使用简体中文。
- 变量名、函数名、文件名使用英文。

## 二、阶段范围

### 第一阶段 MVP

第一阶段目标是跑通“主角角色卡生成闭环”，完成可用性、数据闭环和导出能力。

**包含功能**：
- 寻人启事：用户输入自然语言角色描述。
- 侧写与搜证：剪影选择、反例排除、碎片叙事确认、私密日记破译。
- 世界书创建：世界碎片初探、世界勘探深挖、世界书条目编辑与联想。
- 开场白生成：不同用户身份下生成多个开场白变体，并支持微调。
- 终审对话：多面试官对话、极压测试、日记对话，基于 AI 生成问卷与角色回答形成文本对话结果。
- 导出档案：导出格式化 JSON；首发支持将格式化 JSON 写入 PNG `tEXt` 数据块。
- 本地档案库：独立页面，支持查看、继续编辑、复制为新版本、删除、重新导出。
- 本地持久化：使用 Dexie.js 封装 IndexedDB，保存设置、档案、历史记录、生成结果和未完成进度。

**第一阶段不包含功能**：
- 美化代码生成与正则绑定器。
- 侦探助手笔记。
- 关系网与配角微型寻人。
- 侧写报告导出。

### 第二阶段增强

**美化功能**：为 SillyTavern 聊天消息生成状态栏、气泡、背景等 CSS/HTML/JS 美化代码，并最终打包进单个角色卡 JSON。  
**侦探助手笔记**：展示 AI 推理步骤、调用记录、Token 消耗和耗时概览。  
**关系网与配角**：围绕主角生成关联 NPC，并通过微型寻人流程确认配角设定。  
**侧写报告**：生成一份纪念性质的 HTML 页面，总结本次“寻人”过程，包括初始描述、关键选择、角色档案演化、世界书摘要和终审结果。该报告不是角色卡必需数据，第一阶段不做。

## 三、产品理念与核心隐喻

产品将用户的“角色创建需求”包装为一张**寻人启事**。用户不是填写表单，而是向侧写师搭档描述一个记忆中的人。系统扮演搭档，通过**侧写→搜证→世界勘探→初次接触→终审对话→导出档案**的线性流程，从模糊侧写推进到可导出的角色卡。

第一阶段主流程包含六个主要步骤：寻人启事、侧写、世界书、开场白、终审、导出。步骤间线性推进，但允许随时回溯修改。全局可访问的**角色档案面板**是整场探案的“证据墙”，AI 每一步的推理结果都会更新到这里，用户拥有最终编辑权。

## 四、LLM 接入方式

系统提供两种模型接入方式。主页面检测当前是否存在可用 API；没有可用 API 时，引导用户激活预置模式或进入设置页配置自有接口。

### 方式一：预置调用模式

由项目方通过环境变量配置默认调用地址和密钥，用户无需自行填写 API Key，但必须使用激活码激活。

**激活规则**：
- 用户输入激活码后才能启用预置调用模式。
- 每个激活码只能激活一次，激活后立即作废。
- 每个激活码有效期为自激活起 72 小时。
- 激活状态和到期时间保存在本地数据库中。
- 激活校验由可信服务端完成，不能只在前端静态判断。

**安全约束**：
- 真实服务密钥不能直接打包进前端产物。
- 预置调用模式通过受控网关完成请求。
- 前端只保存激活状态、过期时间、可用模型信息和调用统计，不保存真实预置密钥。
- 项目部署在 Vercel，预置调用模式使用 Vercel Functions 实现受控网关。
- 激活码的“一次性使用”“72 小时有效期”和“调用限制”必须由服务端和数据库共同保证，不能依赖浏览器本地状态。

**Vercel 后端实现**：
- 使用 Vercel Functions 提供 `/api/activate` 和 `/api/llm` 两个接口。
- `/api/activate` 负责校验激活码、写入激活记录、返回短期会话凭证和过期时间。
- `/api/llm` 负责校验会话凭证、检查是否过期、注入服务端保存的真实 API Key，并转发到上游模型接口。
- 激活码、激活记录、调用记录存储在 Neon PostgreSQL。
- Neon PostgreSQL 使用唯一索引和事务保证“激活码只能使用一次”和“按激活时间计算 72 小时有效期”。
- 服务端环境变量保存真实上游 API Key，包括 `PRESET_API_KEY`、`PRESET_API_BASE_URL`、`PRESET_MODEL`，这些变量不能以 `VITE_` 开头。
- `/api/llm` 使用流式响应转发上游模型输出，降低用户感知等待时间。
- `/api/llm` 配置 `maxDuration`，函数执行时间覆盖长文本生成场景。
- 用户使用项目方预置 API Key 时，请求必须经过 `/api/llm` 或等价受控网关，不能由浏览器直接请求上游模型接口。
- 浏览器本地直接请求上游模型接口只用于用户自配 API Key 模式。

**激活码管理**：
- 提供服务端脚本 `scripts/create-activation-code.ts` 生成激活码。
- 激活码生成后写入 Neon PostgreSQL。
- 激活码表包含 `code_hash`、`status`、`created_at`、`activated_at`、`expires_at`、`activation_session_id`、`usage_limit`、`usage_count`。
- 数据库存储激活码哈希，不存储明文激活码。
- `/api/activate` 校验激活码哈希。
- 激活成功后将激活码状态改为已使用，并创建 72 小时有效的激活会话。
- `/api/llm` 按激活会话校验过期时间和调用次数。

**后台管理**：
- 后台管理页面与用户前台部署在同一个 Vercel 项目和同一个域名下。
- 后台路由为 `/admin`。
- 后台接口使用 `/api/admin/*`。
- 后台必须有管理员认证，不能通过前端隐藏入口实现保护。
- 管理员认证使用服务端环境变量 `ADMIN_PASSWORD_HASH` 或后续接入第三方登录。
- 后台可关闭或开启项目方预置模型渠道。
- 预置模型渠道关闭后，普通用户无法使用激活码模式，只能配置自己的 OpenAI 兼容 API。
- 后台可生成激活码。
- 后台可查看激活码列表。
- 后台可查看激活码状态，包括未使用、已激活、已过期、已禁用。
- 后台可查看激活时间、过期时间、调用次数、调用上限。
- 后台可禁用激活码或激活会话。
- 后台所有操作写入管理审计日志。

### 方式二：用户自配 API Key

用户在设置界面中配置 OpenAI 兼容接口，配置保存在 Dexie.js 管理的 IndexedDB 中。

**配置项**：
- API Base URL。
- API Key。
- 模型名称。
- 请求参数：温度。

**兼容要求**：
- 仅支持 OpenAI 兼容接口。
- 兼容部分 API 渠道不支持 `system` prompt 的情况。
- 当接口不支持 `system` prompt 时，系统将系统提示词合并进首条 `user` 消息，并在请求适配器中统一处理。
- 无需支持多模型切换，当前项目只保留一个有效模型配置。

**调用反馈**：
- 每次 AI 调用记录 Token 消耗。
- 每次 AI 调用记录模型耗时。
- 第一阶段不做完整侦探助手笔记，但调用统计进入数据层，供第二阶段展示。
- LLM 调用必须通过项目内统一适配层完成，业务页面不得直接调用 OpenAI SDK。

## 五、角色档案与数据原则

### 角色档案形态

第一阶段中，角色档案以 Markdown 作为主要编辑形态。用户看到和编辑的是一个可读、可改、可锁定的 Markdown 档案，而不是复杂表单。

角色卡最终导出为单个 JSON 文件。该 JSON 严格对齐 Character Card V3，角色卡主体、世界书条目、开场白、美化代码、终审摘要等最终都写入同一个 JSON 文件。具体字段结构由 `docs/character-card-v3-schema.json` 定义。第一阶段导出格式化、可演进的中间 JSON 结构；正式 schema 确认后，导出结构对齐 SillyTavern 可直接导入的最终结构。

### Character Card V3 Schema 文件

正式 schema 使用 JSON Schema 格式，文件路径为 `docs/character-card-v3-schema.json`。`docs/character-card-v3-schema.example.json` 只作为占位参考文件，不参与运行时校验。

正式 schema 包含：
- 顶层字段，例如 `spec`、`spec_version`、`data`。
- 每个字段的类型、是否必填、默认值和说明。
- 嵌套结构，例如世界书条目、开场白数组、扩展字段。

开发时基于正式 JSON Schema 编写对应的 Zod schema，并由 `characterCardBuilder` 负责从项目内部数据结构生成最终导出 JSON。

### Schema 冻结节点

阶段 11 开始前必须完成 `docs/character-card-v3-schema.json`。

Schema 冻结后执行以下工作：
- `src/features/export/characterCardSchema.ts` 对齐正式 schema。
- `src/features/export/characterCardBuilder.ts` 对齐正式 schema。
- 导出 JSON 测试用例对齐正式 schema。
- PNG `tEXt` 写入测试用例对齐正式 schema。

阶段 11 开始前没有正式 schema 时，只实现中间 JSON 导出，不声明 SillyTavern 直接导入兼容。

### 用户确认事实

用户手动编辑后的内容，后续提示词必须标记为“用户确认事实”。AI 在后续生成中必须遵守这些事实，不得随意覆盖。

**基础规则**：
- AI 生成内容标记为“推测中”。
- 用户手动修改过的内容标记为“用户确认事实”。
- 被锁定的内容不能被后续 AI 自动改写。
- 后续提示词必须明确区分“用户确认事实”“已锁定事实”“AI 推测内容”。

## 六、视觉与素材策略

第一阶段不再考虑固定素材库，所有视觉均使用 UI 方式实现。

**实现原则**：
- 侦探档案室氛围通过排版、纹理、边框、纸张感、印章、便签、网格、阴影和交互动效实现。
- 剪影、档案、碎片、日记页、世界书卡片等均用 CSS 和组件结构表达，不依赖图片素材。
- 不使用真人照片或固定插画素材。
- 视觉组件保留替换素材的扩展空间，但第一阶段不以素材资产为前提。

## 七、用户使用流程与功能页面

### 全局元素

1. **背景氛围**：页面始终保持侦探档案室风格，但通过纯 UI 实现，不依赖素材库。
2. **顶部进度条**：显示六个步骤节点：寻人启事、侧写、世界书、开场白、终审、导出。已完成步骤可点击回溯，点击时提示用户确认是否返回修改。
3. **侧边栏：角色档案**：
   - 以 Markdown 编辑器和结构化摘要并存的方式展示当前角色档案。
   - 支持字段级或段落级锁定。
   - 用户可在任意时刻手动编辑内容。
   - 手动编辑内容标记为“用户确认事实”。
   - 锁定内容不会被后续 AI 自动修改。
4. **API 可用性检测**：
   - 主页面显示当前 API 是否可用。
   - API 不可用时，引导用户激活预置调用模式或配置自有 OpenAI 兼容接口。

### 步骤一：张贴寻人启事（Step1_Post）

**页面功能**：用户输入自然语言角色描述。

**交互**：
- 一个大型文本输入框，引导用户描述脑海中的角色。
- 提交按钮。
- 用户输入过于简单时，系统温和追问一句补充问题，但不强制。

**系统处理**：
- 将用户输入发送至 LLM，并附带侧写师系统提示。
- LLM 返回对描述的结构化提取结果。
- 提取结果写入角色档案 Markdown，标记为“推测中”。
- 自动进入侧写步骤。

---

### 步骤二：侧写与搜证（Step2_Profile）

**页面功能**：通过四个互动子步骤，从模糊侧写到精准锁定角色的核心人格、行为模式和深层秘密。四个子步骤在同一页面内以卡片流或选项卡形式展现。

#### 2.1 剪影选择

- **展示**：三张由 UI 绘制的模糊人物剪影，每张下配一句由 AI 生成的内心独白，反映不同行为模式。
- **用户动作**：点击最有共鸣的一张剪影。
- **系统处理**：根据选择锁定角色的基础行为模式，写入角色档案。

#### 2.2 反例排除

- **展示**：三张 UI 化“假想档案”，每张角色都看似符合描述，但带有明显不可接受的特质。
- **用户动作**：将最不想要的一个角色拖入“档案粉碎机”区域，或点击排除按钮。
- **系统处理**：从侧写中剔除对应特质，并强化其对立面，更新角色档案。

#### 2.3 碎片叙事确认

- **展示**：三个由 AI 生成的极短叙事碎片，以 UI 卡片呈现。
- **用户动作**：选择最像“TA 本该存在的样子”的一个碎片。
- **系统处理**：提取该碎片中的行为逻辑，写入角色行为准则、背景故事或性格细节。

#### 2.4 私密日记破译

- **展示**：一张 UI 化日记页，内容由 AI 基于目前侧写生成，关键部分被墨迹遮罩遮挡。
- **用户动作**：点击被涂抹区域，从三个选项中猜测被涂掉的内容。
- **系统处理**：揭示角色的核心秘密或创伤，写入角色档案的核心矛盾，并锁定该内容。

**完成后**：进入世界书创建。

---

### 步骤三：创建世界书（Step3_World）

**页面功能**：创建角色所处的世界观。第一阶段不包含关系网与配角功能。

#### 3.1 世界碎片初探

- **展示**：3-4 张 UI 化“世界碎片”卡片，例如路牌、便签、报纸片段、档案页。
- **用户动作**：
  - 点击卡片上的文字直接修改。
  - 回答每个碎片配套的问题。
  - 支持跳过此步并在后续补充。
- **系统处理**：AI 根据角色档案和碎片信息生成初步世界书条目，填入角色档案。

#### 3.2 世界勘探深挖

- **展示**：所有已生成的世界观条目以可翻转卡片阵形式排列。
- **用户动作**：
  - 直接编辑卡片文本。
  - 点击“深度挖掘”，由 AI 针对该条目生成 3 个细化问题，用户回答后追加到条目。
  - 拖动两个条目卡片重叠，由 AI 生成新的关联条目。
  - 点击“从当前条目联想”，由 AI 推理可能缺失的条目，用户选择是否添加。
- **系统处理**：所有修改实时同步到角色档案的世界观部分，并进入本地历史记录。

**多任务说明**：项目不支持多个角色项目并行创作；同一项目内，世界书条目生成支持多个任务并行，便于用户一次生成多个候选条目。

**完成后**：进入开场白生成。

---

### 步骤四：开场白生成（Step4_Greeting）

**页面功能**：让用户以不同身份“偶遇”角色，从多角度预览和微调开场对话。

**交互**：
- **身份选择器**：提供“陌生人”“委托人”“老友”“敌人”等预设身份。
- **分屏预览**：每选择一个身份，下方刷新对应的开场白文本和环境氛围描述。
- **开场白卡片**：每个身份下提供 2-3 个变体，用户可点选最满意的一条。
- **微调面板**：可修改语气热烈程度、长度、必须包含的要素。
- **用户确认**：选定即锁定开场白，写入角色档案。

**系统处理**：AI 根据当前角色档案和世界书生成不同身份下的多版本开场白。开场白生成在同一项目内支持多个候选任务并行。

---

### 步骤五：终审对话（Step5_Trial）

**页面功能**：作为导出前的质量闸门，通过多种对话测试检验角色是否 OOC（角色性格偏差）。第一阶段以“生成好的文本对话结果”为主，不做复杂实时聊天交互。

**命名调整**：
- “多面试官审讯”更名为“多面试官对话”。
- “日记审讯”更名为“日记对话”。
- 页面文案减少压迫感，保留测试角色一致性的功能目标。

**生成机制**：
- 每次测试调用两轮 AI。
- 第一轮 AI 根据角色档案、世界书、核心矛盾和测试模式生成一份测试问卷。
- 第二轮 AI 扮演已经生成的角色，对测试问卷逐项回答。
- “正式回复”和“内心独白”在第二轮请求中同时生成。
- 最终结果以文本对话形式呈现给用户。

**可用测试模式**：

1. **多面试官对话**
   - 三位固定风格的虚拟面试官从不同角度提问。
   - AI 先生成问卷，再让角色完成回答。
   - 用于观察角色在不同关系压力下的表达稳定性。

2. **极压测试**
   - AI 问卷从日常话题逐步升级到信念质疑、创伤触碰和珍视之人相关问题。
   - OOC 风险由模型自评，通过提示词明确边界。
   - 前端根据模型返回的风险标注和角色档案关键事实进行基础比对，并高亮可能存在 OOC 风险的句子。

3. **日记对话**
   - AI 问卷引用私密日记或核心矛盾内容，要求角色回应过去誓言与当前行为之间的张力。
   - 用于检验人物弧光是否合理，而不是制造强压迫感。

4. **沉默证词**
   - 每个问答结果同时展示角色正式回复和内心独白。
   - 内心独白用于检验角色是否有合理反差。

**历史记录与回溯**：
- 终审结果长期保存。
- 用户可多次运行终审测试。
- 重新测试时可清空当前测试结果并重新生成。
- 用户可根据历史记录从任意节点重新开始，形成一个新版本。
- 测试过程中若发现不满意，可返回侧写、世界书或开场白步骤调整；已有角色档案数据保留。

**完成条件**：用户确认角色行为符合预期，点击“通过终审”进入导出。

### 步骤六：导出档案（Step6_Export）

**页面功能**：将成果导出为角色卡文件。

**用户操作**：
- 点击“导出角色卡”按钮。
- 可填写角色卡备注或版本号。
- 选择导出格式：格式化 JSON 文件或内嵌 JSON 的 PNG 图片。
- 系统生成文件并触发浏览器下载。

**导出内容**：
- 严格朝 Character Card V3 对齐的格式化 JSON。
- 世界书条目数据。
- 开场白数据。
- 终审对话历史摘要。
- 第二阶段完成后，美化代码也打包进同一个 JSON 文件。
- 首发支持将格式化 JSON 写入 PNG `tEXt` 数据块。

**第一阶段导出边界**：
- 第一阶段导出正确的中间 JSON 文件。
- 正式 schema 确认后，完善为 SillyTavern 可直接导入使用的最终结构。
- PNG 写入功能首发实现，内部 JSON 结构随 Character Card V3 结构确认后继续演进。

**完成后**：本地档案库中保存一份副本，方便后续继续编辑、复制为新版本或重新导出。

## 十一、核心模块设计约束

### 可复用模块与组件规范

第一阶段开发时必须沉淀可复用模块，避免每个步骤页面各自实现一套相似逻辑。页面层只负责组合和路由衔接，跨步骤共享的交互、状态和服务必须放入 `features`、`shared`、`db` 或 `schemas` 中。

**必须复用的组件**：
- `StepProgress`：顶部流程进度条，支持当前步骤、已完成步骤、回溯确认。
- `DossierPanel`：全局角色档案侧栏，所有主流程页面共用。
- `DossierEditor`：角色档案 Markdown 编辑器，支持锁定、用户确认事实和自动保存。
- `ApiStatusBadge`：展示当前 API 可用性、激活状态和过期状态。
- `GenerationButton`：统一 AI 生成按钮状态，包含加载、取消、重试和错误反馈。
- `GeneratedResultCard`：展示 AI 生成候选内容，适用于侧写、世界书、开场白和终审结果。
- `ChoiceCard`：选择类交互共用卡片，适用于剪影选择、反例排除、碎片选择和开场白候选。
- `EditableCard`：可编辑内容卡片，适用于世界书条目、假想档案和碎片文本。
- `ConfirmDialog`：回溯、删除、清空重测、覆盖内容等确认操作共用。
- `EmptyState`：无档案、无历史、无 API、无生成结果等空状态共用。
- `HistoryTimeline`：展示历史快照、生成记录和从节点重新开始入口。
- `TokenUsageMeter`：展示 Token 消耗和模型耗时，第一阶段用于生成记录，第二阶段可复用于侦探助手笔记。

**必须集中管理的状态模块**：
- `flowStore`：当前步骤、已完成步骤、回溯目标和步骤跳转状态。
- `dossierStore`：角色档案 Markdown、锁定状态、用户确认事实和自动保存状态。
- `generationStore`：当前生成任务、并行任务、加载状态、错误状态和取消控制器。
- `settingsStore`：API Base URL、模型、温度、是否支持 `system` prompt、自配 Key 状态。
- `activationStore`：激活码状态、激活时间、过期时间和预置模式可用性。
- `uiStore`：侧边栏开合、当前面板、弹窗和主题相关 UI 状态。
- `historyStore`：历史快照、版本复制和从某节点恢复。
- `exportStore`：导出格式、导出备注、最近导出记录和导出状态。

**必须复用的服务层**：
- `llmClient`：统一 AI 调用入口，业务页面不得直接调用 OpenAI SDK。
- `promptAdapter`：统一处理 `system` prompt 兼容、用户确认事实注入和锁定内容约束。
- `usageTracker`：记录 Token、耗时、模型、调用来源和错误摘要。
- `historyService`：创建快照、恢复快照、复制新版本。
- `characterCardBuilder`：从档案、世界书、开场白、终审记录和备注构建导出 JSON。
- `pngTextWriter`：写入和校验 PNG `tEXt` 数据块。
- `autoSaveService`：编辑后防抖保存到 IndexedDB。

**复用边界原则**：
- 页面组件只做页面组合、路由参数读取和局部布局。
- 业务流程逻辑放在对应 `features` 目录中。
- 数据库读写只能通过 `db/repositories` 或领域 service 完成。
- 提示词只能放在 `prompts` 或对应 `features/*/*Prompts.ts` 中，不能散落在组件里。
- Zod schema 集中维护，不能在页面中临时定义导出结构。
- AI 生成任务必须产生 `GenerationTask` 记录，便于历史回溯、Token 统计和错误排查。

### LLM 适配层

业务层只能调用 `llmClient` 暴露的统一方法，例如 `generateProfile`、`generateWorldEntries`、`generateGreetings`、`generateTrialQuestionnaire`、`answerTrialQuestionnaire`。

适配层必须处理：
- 预置调用模式和自配 API Key 模式切换。
- 不支持 `system` prompt 的接口兼容。
- 非流式和流式响应。
- Token 用量与耗时统计。
- JSON 提取、Zod 校验和错误提示。
- 请求取消和重复生成任务的状态同步。

### 本地数据层

Dexie 表结构第一阶段包含：
- `settings`：API 设置、模型设置、界面偏好。
- `activations`：激活状态、激活时间、过期时间、可用模型信息。
- `adminSettings`：后台管理相关的前端状态缓存，不保存管理员密钥。
- `projects`：当前角色项目、角色档案 Markdown、当前步骤、导出状态。
- `histories`：从任意节点重新开始所需的历史快照。
- `generations`：每次 AI 生成任务、输入摘要、输出、Token 消耗、耗时、错误信息。
- `exports`：导出版本、JSON 内容摘要、PNG 生成信息、备注。

Neon PostgreSQL 表结构第一阶段包含：
- `activation_codes`：激活码哈希、状态、创建时间、激活时间、过期时间、调用上限。
- `activation_sessions`：激活会话、过期时间、调用次数、禁用状态。
- `admin_audit_logs`：管理员操作记录。
- `model_channel_settings`：项目方预置模型渠道开关状态。

### 角色档案

角色档案以 Markdown 为用户编辑主体，内部保存元数据：
- 哪些段落来自 AI 推测。
- 哪些段落是用户确认事实。
- 哪些段落或字段被锁定。
- 哪次生成任务修改了哪些内容。

### Markdown 档案与元数据映射

第一阶段采用“块级 ID + 旁路元数据”的方案。Markdown 文本保持可读、可编辑，不在正文里塞大量隐藏标记；系统在内部用块级 ID 追踪每个段落或字段的来源、锁定状态和用户确认状态。

**映射规则**：
- 每个可追踪段落称为一个 `DossierBlock`。
- 每个 `DossierBlock` 拥有稳定的 `blockId`，例如 `blk_core_personality_001`。
- Markdown 正文按固定标题组织，例如 `## 核心人格`、`## 外貌特征`、`## 背景故事`、`## 核心矛盾`、`## 说话风格`、`## 世界观`、`## 开场白`。
- 元数据不直接依赖文本位置，而是保存 `blockId`、所属 section、内容 hash、来源、锁定状态、最后修改来源和关联生成任务。
- 用户编辑某个块后，该块标记为 `user_confirmed`。
- 用户锁定某个块后，后续 AI 更新必须跳过该块。
- 用户大幅改写导致 hash 不匹配时，系统以当前用户文本为准，并保留 `user_confirmed` 状态。

**内部结构**：

```ts
type DossierBlockMeta = {
  blockId: string;
  section: string;
  contentHash: string;
  source: "ai_inferred" | "user_confirmed";
  locked: boolean;
  updatedByGenerationId?: string;
  updatedAt: string;
};
```

该方案使用块级元数据追踪事实来源和锁定状态，Markdown 正文保持干净可读。编辑器升级为块编辑器时，不改变导出结构。

构建 Character Card V3 JSON 时，由 `characterCardBuilder` 从角色档案 Markdown、世界书条目、开场白、终审记录和导出备注组装，禁止直接拼接字符串。

### PNG 写入

PNG 角色卡导出使用用户上传图片作为载体。用户上传 JPG、PNG、WebP 等浏览器可读取的图片格式后，系统先在浏览器端转换为 PNG，再写入 Character Card JSON。

用户上传图片要求：
- 支持 JPG、PNG、WebP。
- 上传图片仅用于 PNG 角色卡导出的视觉载体。
- 上传图片不进入界面素材库。
- 上传图片转换后的 PNG 由 `pngTextWriter` 写入 `tEXt` 数据块。
- 原始上传文件和转换后的 PNG 可作为导出版本记录的一部分保存在 IndexedDB。

PNG 写入流程：
- 用户上传图片。
- 浏览器读取图片并绘制到 Canvas。
- Canvas 导出 PNG 二进制。
- 将 Character Card JSON 使用 `JSON.stringify(data, null, 2)` 格式化。
- 将格式化 JSON 写入 PNG `tEXt` 数据块。
- 下载前可进行一次读取校验，确认写入内容可被重新提取。

### 历史与版本

用户不能并行编辑多个角色项目，但可基于历史记录从任意节点重新开始。实现上保存快照，而不是只保存最终 Markdown。

历史快照至少包含：
- 当前步骤。
- 角色档案 Markdown。
- 锁定状态与用户确认事实元数据。

## 十二、环境变量与配置

前端环境变量只保存可公开信息，不能保存需要保密的真实密钥。

```txt
VITE_APP_NAME=MOI
VITE_PRESET_API_BASE_URL=
VITE_ACTIVATION_ENDPOINT=
VITE_DEFAULT_MODEL=
```

预置调用模式使用项目方密钥时，`VITE_PRESET_API_BASE_URL` 指向受控网关。网关负责校验激活状态、注入真实密钥、转发请求和限制滥用。

Vercel 服务端环境变量：

```txt
PRESET_API_BASE_URL=
PRESET_API_KEY=
PRESET_MODEL=
ACTIVATION_TOKEN_SECRET=
ADMIN_PASSWORD_HASH=
DATABASE_URL=
```

用户自配 API Key 存储在 IndexedDB 中，只在本地浏览器使用。设置页明确提示浏览器端保存密钥的风险。

## 十四、用户体验亮点总结

1. **沉浸式侦探叙事**：从“寻人启事”到“终审对话”，每一步都像在逐步确认一个本来就存在的人。
2. **零门槛侧写系统**：用选择、排除、碎片和日记破译替代表单，让新手也能表达复杂角色。
3. **强掌控角色档案**：Markdown 档案、用户确认事实、锁定机制和回溯历史共同保证用户拥有最终编辑权。
4. **可深挖的世界书**：世界观条目可编辑、可关联、可继续联想，避免世界书只是一次性生成文本。
5. **低压但有效的终审对话**：通过问卷生成与角色回答，检查 OOC 风险，同时减少“审讯”措辞带来的压迫感。
6. **本地优先的数据体验**：Dexie.js 管理所有本地设置与档案，支持断点续连、历史记录和版本复制。
7. **清晰的阶段扩展**：第一阶段先完成主角角色卡闭环，第二阶段再扩展美化、侦探助手笔记、配角关系网和侧写报告。

---

《来岛上》的目标不仅是成为一个工具，更是一个让创作者与角色“相遇”的媒介。第一阶段必须保证主流程完整、数据结构稳定、AI 调用可靠、本地保存可信、导出路径可演进。
